# テスト計画（MVP）

## 方針（優先順位）
- **統合テスト重視**: 構造変更があっても統合テストが通れば自信を持てる状態を目指す
- 単体テストは補助: 仕様の境界条件やエラー処理の最小限を担保する

## 統合テストの戦略
- `git init` で “ローカルのダミーリモート” を作り、それを remote URL として扱う
- fetch/prune などはローカルでも成立するようにする
- DB 等の外部依存がない前提なので、実行時間は過度に最適化しない
- 重要フローは **統合テストを最優先で整備**する

## DoD（MVP）
- `go test ./...` が通る
- 主要コマンド（repo get、new/add/status/rm）が統合テストで一通り動く
